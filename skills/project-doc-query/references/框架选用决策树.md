# 框架选用决策树（Framework Selection Tree）

> 当用户意图为"决策建议"时，本决策树确定优先使用哪个框架。

## 决策树

```
用户问题是什么类型？
├─ 评审/交付/里程碑安排
│   └─ 【PMP 框架层 · 进度管理】+【PRINCE2 落地层 · 管理阶段边界】
├─ 范围/变更/基线
│   └─ 【PRINCE2 落地层 · 变更主题】+【PMP 框架层 · 综合/范围/变更】
├─ 质量/测试/缺陷
│   └─ 【系统分析师 实务层 · 测试与维护】+【PMP 框架层 · 质量管理】+【PRINCE2 落地层 · 质量主题】
├─ 风险
│   └─ 【PMP 框架层 · 风险管理】+【PRINCE2 落地层 · 风险主题】
├─ 资源/成本/采购
│   └─ 【PMP 框架层 · 资源/成本】+【PRINCE2 落地层 · 商业论证主题】
├─ 需求
│   └─ 【系统分析师 实务层 · 需求分析】+【PMP 框架层 · 范围】
├─ 架构/设计
│   └─ 【系统分析师 实务层 · 系统设计】+【PMP 框架层 · 范围/进度】
├─ 实施/部署/运维
│   └─ 【系统分析师 实务层 · 系统实施与运维】+【PMP 框架层 · 执行】+【PRINCE2 落地层 · 交付主题】
├─ 收尾/验收
│   └─ 【PMP 框架层 · 收尾过程组】+【PRINCE2 落地层 · 持续业务验证】
└─ 沟通/汇报/会议
    └─ 【PMP 框架层 · 沟通管理】+【PRINCE2 落地层 · 计划主题】
```

## 三层框架的角色定位

| 框架 | 层次 | 解决什么 |
|---|---|---|
| **PMP** | 框架层 | "管理体系长什么样"（5 过程组 / 10 知识领域） |
| **PRINCE2** | 落地层 | "具体每一步怎么做"（7 流程 / 7 主题） |
| **系统分析师** | 实务层 | "软件工程中的实际工程问题"（5 大模块：规划/需求/设计/测试维护/信息化） |

> 三者**不是替代关系**，而是**叠加**。
> 例：质量类问题既要从 PMP 质量管理看，又要从 PRINCE2 质量主题落地，还要从系统分析师的测试模块看实务。

## 输出格式

每条决策建议**首行**强制：

```
【框架：{PMP|PRINCE2|系统分析师} · {框架层|落地层|实务层}】
```

完整叠加时按"框架层 → 落地层 → 实务层"顺序输出。
