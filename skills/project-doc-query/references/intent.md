# intent.md · 流程澄清（intent 维度 · 5 子项）

> 配套 `intent-clarification/SKILL.md` 使用。
> owner skill：project-doc-query
> 场景分类：A. 流程澄清

## 概述

本文件涵盖**流程启动时**的 5 个基础澄清项。当任何 project-doc skill 启动并需要和用户对齐意图时，按本文件逐项询问。

5 个子项：
1. **intent_fact_or_decision** — 事实查询 vs 决策建议
2. **scope_project_or_industry** — 项目相关 vs 行业通用
3. **project_root** — 项目根目录
4. **doc_type** — 目标文档类型
5. **action_intent** — 动作意图（查询 / 生成 / 更新）

---

## 1. intent_fact_or_decision（事实 vs 决策）

### 询问话术

```markdown
在回答您的问题前，请先澄清意图：

您问的是：
(a) **事实/数据查询**（如"测试方案什么时候提交？"）— 我会直接给数据 + 来源
(b) **决策建议**（如"测试方案应该覆盖哪些内容才算合格"）— 我会用 PMP/PRINCE2/系统分析师 框架给建议
```

### 关键词分类

| 关键词 | 分类 |
|---|---|
| 什么时候 / 谁负责 / 多少 / 在哪 / 是什么 | 事实查询 (a) |
| 怎么 / 应该 / 建议 / 是否 / 哪种 | 决策建议 (b) |

### 默认行为

- 用户没说 → 默认 **(a) 事实查询**
- 用户说"建议"但没指定项目 → **(b)** + **注明**"基于行业通用做法，非本项目特定"

---

## 2. scope_project_or_industry（项目 vs 行业）

### 询问话术

```markdown
范围上，您希望：
(i) **仅看与本项目相关的答案**（基于策划表/需求/方案等已有资料）
(ii) **了解行业通用做法**（可能引入框架知识）
```

### 默认行为

- 用户没说 → 默认 **(i) 项目相关**

---

## 3. project_root（项目根目录）

### 询问话术

```markdown
请告诉我项目根目录的绝对路径：
（如 D:\项目文档\202410-C0008-...）
```

### 解析

- 项目根 = 含 `01_项目策划/` 等子目录的目录
- 父目录 = `<用户工作根>`，用于建 `.project/` 目录
- 项目号 = 解析项目根目录名前 2 段（如 `202410-C0008`）

### 错误处理

- 路径不存在 → 提示用户检查
- 多项目 → 让用户选

---

## 4. doc_type（目标文档类型）

### 询问话术

```markdown
请选择目标文档类型：

1. 售前方案
2. 需求说明书
3. 概要设计说明书
4. 详细设计说明书
5. 实施方案
6. 测试方案
7. 测试报告
8. 验收报告
9. 实施部署方案
10. 培训方案
11. 其他过程文档
```

### 用途

- query 阶段：选择 reference 模板加载
- outline 阶段：选择章节大纲模板
- write 阶段：选择写入位置 + 格式范本

---

## 5. action_intent（动作意图）

### 询问话术

```markdown
您希望做什么动作：

1. **查询** — 查项目里的事实/数据（"什么时候交""谁负责"）
2. **生成** — 从 0 创建一份新文档
3. **更新** — 在已有文档上补充/修改某些章节
4. **删除** — 删除某份文档（罕见）
```

### 默认行为

- 用户没说 → 默认 **1. 查询**（最轻量）

---

## 复合问题处理

如果用户问的**既是事实又是决策**（如"测试方案什么时候交？应该怎么写？"）：

- **必须**分两轮问
- **禁止**混答（违反"事实/决策隔离"原则）

```markdown
您问了两个问题，我会分两次回答：

Q1: 测试方案什么时候提交？→ 这是事实查询 (a)
   答案：策划表 §评审计划 标注 K3 节点 2025-12-10

Q2: 测试方案应该怎么写？→ 这是决策建议 (b)
   答案：基于系统分析师 实务层（测试与维护）框架，应覆盖...
```

## 记录到日志

每次问完一项追加到 `clarification_log.md`：

```bash
python manage_project_log.py append-clarification \
    --project-id <ID> --work-root <ROOT> \
    --dimension "A.intent" --item "intent_fact_or_decision" \
    --question "您问的是事实还是决策？" --answer "事实" \
    --source "用户口述" --asked-by "project-doc-hub"
```

5 个子项的 dimension 标识统一为 `A.intent`，item 分别为：
- `intent_fact_or_decision`
- `scope_project_or_industry`
- `project_root`
- `doc_type`
- `action_intent`

---

## 反模式

| 反模式 | 后果 |
|---|---|
| 跳过澄清直接给"应该/建议" | 用户可能只想看事实，被推荐误导 |
| 用"决策建议"包装"事实" | 误导用户以为是既定结论 |
| 没有澄清就引用框架 | 用户不知道答案基于哪个体系 |
| 5 个子项一锅问 | 用户难以回答 |
| 复合问题混答 | 违反隔离原则 |
