# 意图澄清提问模板（Intent Clarification）

> `project-doc-query` 在回答前**必须**先做意图澄清。本模板提供标准提问话术。

## 提问脚本

```
在回答您的问题前，我需要先澄清两点：

1) 您问的是：
   (a) 事实/数据查询（如"测试方案什么时候提交"）—— 我会直接给数据 + 来源
   (b) 决策建议（如"测试方案应该覆盖哪些内容才算合格"）—— 我会按 PMP/PRINCE2/系统分析师 框架给建议

2) 您希望：
   (i) 仅看与本项目相关的答案（基于策划表/需求/方案等已有资料）
   (ii) 想了解行业通用做法（可能引入框架知识）

请告诉我：
- 您的意图是 (a) 还是 (b)？
- 范围是 (i) 还是 (ii)？
```

## 意图分类

| 关键词（中文） | 关键词（英文） | 分类 |
|---|---|---|
| 什么时候 / 谁负责 / 多少 / 在哪 / 是什么 | when / who / how many / where / what | 事实查询 (a) |
| 怎么 / 应该 / 建议 / 是否 / 哪种 | how / should / recommend / is it / which | 决策建议 (b) |

## 默认行为

- 用户没说 → 默认按 (a) 事实查询 + (i) 项目相关
- 用户说"建议"但没说项目 → 走 (b) + (ii)，**但必须**注明"基于行业通用做法，非本项目特定"
- 复合问题（既问事实又问建议）→ 分别按 a/b 处理，**禁止混答**

## 严禁

- 跳过澄清直接给"应该/建议"（用户可能只想看事实）
- 把"决策建议"包装成"事实"（误导用户以为是既定结论）
- 没有澄清就引用框架（用户不知道答案是基于哪个体系）
