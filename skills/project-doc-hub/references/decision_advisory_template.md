# 决策与意见生成模板（Decision Advisory Template）

> 在 `project-doc-write` 阶段套件必须**给出**决策与意见；本模板规定格式与来源。
> **严禁凭空假设**——所有决策与意见必须能从项目已有资料中追溯。

## 决策与意见章节结构（推荐放在文档末或独立附录）

```markdown
## 决策与意见

### 1. 范围决策
- 决策点：xxx
- 依据：策划表《范围管理》子标签 + 合同交付物清单 + 需求说明书 vN
- 建议：……
- 风险：……

### 2. 进度决策
- 决策点：xxx
- 依据：策划表《里程碑》子标签 + 周报 N
- 偏差：计划 X 天，实际 Y 天（偏差 Z%）
- 建议：……

### 3. 质量决策
- ……

### 4. 风险决策
- ……

### 5. 资源/成本决策
- ……
```

## 决策与意见的来源（仅允许从这些数据源提炼）

| 数据源 | 可提炼的决策类型 |
|---|---|
| 策划表《风险登记册》 | 风险应对决策、规避/转移/减轻/接受 |
| 策划表《汇总损益分析》 | 成本/资源调整决策 |
| 策划表《假设与约束》 | 范围/边界决策 |
| 策划表《里程碑》vs 周报实际进度 | 进度偏差决策、变更基线 |
| 合同交付物清单 vs 实际交付 | 范围/验收决策 |
| 需求说明书 vs 设计文档 | 需求实现度、技术方案决策 |
| 设计文档 vs 测试用例 | 覆盖率、测试深度决策 |
| 客户/监理邮件中的新要求 | 变更决策、范围蔓延识别 |

## 严禁的"决策"（红线）

- ❌ "基于行业最佳实践" 类的无依据建议
- ❌ "通常项目应该……" 类的通用知识
- ❌ 没有任何数据来源的"建议"
- ❌ 编造合同金额、日期、人名

## 多框架叠加（PRINCE2 + PMP + 系统分析师）

每条决策与意见回答中**强制输出**：

```
【框架：{PMP|PRINCE2|系统分析师} · {框架层|落地层|实务层}】
```

例：

- "建议将测试覆盖率从 70% 提升到 85% 【框架：PMP · 框架层 — 质量管理知识领域】，依据策划表《质量目标》子标签中'功能测试覆盖率 85%'的设定"
- "建议走变更控制流程【框架：PRINCE2 · 落地层 — 变更主题】，依据策划表《变更控制》子标签的分级规则"
- "建议补充非功能需求【框架：系统分析师 · 实务层 — 需求分析模块】，依据需求说明书中'性能'章节缺失"
