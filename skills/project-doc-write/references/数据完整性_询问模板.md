# write Step 4.2.5 数据完整性询问模板

> 当目标文档是"数据驱动"型（测试报告 / 测试总结 / 验收报告 / 实施总结等），
> write 必须用本模板的话术主动询问用户。

---

## 触发条件

满足以下任一条件即触发：

1. 目标文档是以下之一：
   - 测试报告
   - 测试总结
   - 验收报告
   - 实施总结
   - 试运行报告
   - 任何章节含"用例数/缺陷数/性能指标/覆盖率/通过率"等数据字段的文档
2. 这些数据字段在项目资料中（策划表 / 需求 / 合同 / 邮件 / 周报）**无证据**

---

## 询问话术（直接复制使用）

### 模板 A：首次询问（章节级）

> **关于文档《[文档名]》的数据完整性询问**
>
> 以下章节需要实际数据，但当前项目资料中无证据：
>
> | 章节 | 所需数据 | 当前项目资料中是否有 |
> |---|---|---|
> | [章节号] [章节名] | [字段 1] / [字段 2] / ... | ❌ 无 |
> | [章节号] [章节名] | [字段 1] / [字段 2] / ... | ❌ 无 |
>
> **请选择**：
> 1. **提供数据**（请直接告诉我每个字段的具体值）
> 2. **保持"待补"状态**（按 `references/文档内容_净化规则.md §3.1` 写法，不带"—"占位符）
> 3. **暂不生成**该文档（等数据齐了再写）

### 模板 B：逐章询问（细粒度）

> 文档《[文档名]》第 [X] 章"[章节名]"需要以下数据：
>
> - [字段 1]：
> - [字段 2]：
> - [字段 3]：
>
> 请问如何处理？
> 1. 逐项提供（请按上面列表顺序回答）
> 2. 全部"待补"
> 3. 跳过本章

### 模板 C：再次确认（用户已选"保持待补"）

> 您已选择"保持'待补'状态"。该章节将按以下格式写入：
>
> ```markdown
> ## [X.Y] [章节名]
>
> **待补**：本章需项目组在 [预计补全时间] 后回填实际数据。
>
> 数据源建议：
> - [来源 1]
> - [来源 2]
>
> 建议字段：[字段列表]
> ```
>
> 确认？或修改措辞？

## 询问时机

| 场景 | 时机 |
|---|---|
| 文档是数据驱动型 | Step 4.1 写每章前询问该章所需数据 |
| 文档是过程文档型（实施方案/方案设计） | 不询问（无数据章节） |
| 用户中途说"先出个 V1.0" | 询问一次"哪些章节'待补'，哪些章节可填" |

---

## 询问原则

1. **不擅自编造**：所有数据必须用户给或项目资料里有
2. **不写"—"占位**：表格行空着不如用"**待补**"段
3. **不重复询问**：同一份文档的同一组数据，询问一次即可
4. **可中断**：用户可说"全部待补"，write 立即按"待补"模式继续

---

## 与"评审稿"的根本区别

| 旧做法（评审稿） | 新做法（待补） |
|---|---|
| 文档状态：评审稿 | 不写状态（用户不说就沉默） |
| > 评审稿阶段尚未填写具体数据 | **待补**：本章需项目组在 [时间] 后回填实际数据 |
| \| 单元测试 \| — \| — \| — \| | **不写空表**，改用"**待补**"段 |
| 编制：张三（策划表没张三） | **不写角色表**（策划表没就沉默） |

> 详细净化规则见 `references/文档内容_净化规则.md`
