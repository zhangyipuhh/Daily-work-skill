# 软件工程文档 章节填写规范

> 本规范为 `project-doc-write` 阶段的章节填写标准。

## 通用原则

1. **每章首句说明本章目的**（1-2 句）
2. **每章末给"小结"或"待确认事项"**（如有）
3. **章节标题用阿拉伯数字编号**（1、1.1、1.1.1）
4. **表格必须有表头和单位**
5. **插图必须有图编号和图标题**
6. **引用必须标注文档来源**（如"依据策划表《评审计划》子标签"）

## 各章节类型填写规范

### 引言类章节（1.1/1.2/1.3）

- 写**项目背景、范围、读者、术语**
- 背景：可引用合同/项目立项书
- 范围：明确"做什么/不做什么"
- 读者：列出主要读者角色

### 总体/概述类章节

- 配**一张总体图**（架构图/关系图/流程图）
- 写**3-5 条核心要点**
- 用列表/表格，不长篇大论

### 设计/方案类章节

- 先说"为什么这么选"（方案对比）
- 再说"具体怎么做"（步骤/参数/配置）
- 最后说"风险与备选"（Plan B）

### 计划/进度类章节

- 表格列出：阶段/起始/结束/交付物/责任人
- 与策划表《里程碑》《WBS》子标签**对得上**
- 偏差要标注

### 测试类章节

- 测试用例：编号/模块/输入/步骤/预期/实际
- 缺陷：编号/等级/描述/状态/责任人
- 覆盖率：明确度量方法

### 风险类章节

- 风险：编号/描述/概率/影响/应对/责任人
- 风险等级：高中低
- 与策划表《风险登记册》对得上

## 表达规范

- 不用"我们/你"，用"项目组/承建方/客户方"
- 时间格式：`YYYY-MM-DD`
- 文件路径：`<项目根>/...` 用相对项目根的路径
- 金额单位：万元（带单位）

## 严禁

- "等"/"等等" 类不明确表述
- "大约"/"大概" 类模糊量
- "应该"/"可能"/"或许" 类无依据表述
- 编造的合同金额、日期、人名
