# no_fabrication.md · 不瞎编规则

> 配套 `intent-clarification/SKILL.md` 的 `<HARD-GATE: NO FABRICATION>` 段使用。
> **本规则适用于所有 project-doc skill**（query / outline / write / workflow）。

## 7 条不瞎编铁律

### 1. 人名

- ❌ 资料中无 → 禁止虚构
- ✅ 资料中有 → 按资料填写
- ❌ 资料中无具体姓名 → 禁止写角色签名表

### 2. 日期

- ❌ 资料中无 → 禁止虚构
- ✅ 资料中有 → 按资料填写
- ❌ 资料中无 → **待补**：<字段名>

### 3. 数字 / 指标

- ❌ 资料中无 → 禁止虚构
- ✅ 资料中有 → 按资料填写
- ❌ 资料中无 → **待补**：<字段名>

### 4. 工具 / 平台 / 软件

- ❌ 项目资料未提 → 禁止虚构（"禅道"/"Git"/"钉钉"等）
- ✅ 资料中有 → 按资料填写
- ❌ 资料中无 → 章节标"由项目组选定"

### 5. 角色签名表

- ❌ 资料中无具体姓名 → 禁止虚构
- ✅ 资料中有 → 按资料填写
- ❌ 资料中无 → **不写角色表**

### 6. 文档状态

- ❌ 用户没说 → 禁止写"评审稿/草稿/正式版"
- ✅ 用户明确说"出 V1.0 初稿" → 写"文档状态：初稿"

### 7. 框架标签（PMP/PRINCE2/系统分析师）

- ❌ 答案本质是系统分析师层面 → 禁止贴 PMP 标签
- ✅ 答案与所选框架对应 → 准确标注
- 不确定时 → **不贴标签**（写"基于 XXX 实务层常识"等不带标签的措辞）

## 严禁的占位符

任何 skill 任何场景禁止单独使用：

```markdown
❌ | XX | — |
❌ | XX | TBD |
❌ | XX | 待定 |       ← 单独使用无说明
❌ | XX | N/A |
❌ XX（无说明）
```

## 推荐的占位符写法

必带说明：

```markdown
✅ | XX | **待补**：<具体字段名> |
✅ | XX | **待补**：用例计划数（策划表未给） |
✅ | XX | 约 100-150 万（**待补**：精确值需项目组提供） |
✅ | XX | 按 [文件] §X 第 Y 行 |
```

## 触发流程

写正文时遇到证据缺失：

1. **立即调** `intent-clarification` 走对应维度（B.data / C.environment / D.document_attr / F.diff）
2. 用户回答"待定" → 重新问"停止 / 提供详细信息"
3. 用户没指定 → **不写**，**不擅自填默认值**
4. 写"**待补**：<字段名>"段（不写"—"）
5. 记录到 `.project/<项目号>/clarification_log.md`

## 严禁的反模式

```markdown
❌ 思考中"我应该问"但继续写
❌ 写完后再补问
❌ 把"用户说新的"当"已确认差异"
❌ 虚构人名/日期/数字/工具
❌ 单独使用"—/TBD/待定"
```

## 写完自检清单

write 流程结束前**必须**自检：

- [ ] 没有人名虚构
- [ ] 没有日期虚构
- [ ] 没有数字虚构
- [ ] 没有工具虚构
- [ ] 没有"评审稿"等用户没说的状态
- [ ] 没有"编制/审核/批准"虚构姓名
- [ ] 表格无"—"单独占位
- [ ] 框架标签准确

**违反任何一条 → 删除重写**，禁止"加注释过关"。
